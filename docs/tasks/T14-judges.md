# T14 — Судьи: композиция, ре-итерации, эскалация

- **Веха:** M3
- **Зависит от:** T06, T09
- **Спека:** SDD FR-6, FR-4.5, FR-11.3; Т-3, Т-6, Т-8

## Скоуп

- `JudgeExecutor`: опциональный LLM-вердикт (отдельный вызов AgentRuntime) + **обязательный**
  детерминированный recheck (`check_*.py` через ScriptExecutor) для судей, блокирующих outward.
- Ингест LLM-вердикта: verdict.json читает движок; при противоречии решает recheck.
- Промпт LLM-судьи изолирует артефакт как данные (анти-injection структура, Т-8).
- Вердикты → `StepRun.verdicts` (SoT), событие `judge.verdict` в аудит; агенту недоступны (Т-3).
- FAIL: накопление `errors.json` (perpetual, по итерациям), ре-итерация целевой фазы с
  `accumulated_errors` в промпте, лимит из FailPolicy (дефолт 3) → эскалация (диалог T12).
- Скрипты судей запускаются по путям IDE-кэша обвязки (интерфейс; сам кэш — T18,
  до его готовности — временно из проекта с TODO-маркером).

## Вне скоупа

- Кэш обвязки и hash-манифест (T18), dry-run (T21).

## Приёмка

- [x] LLM-PASS + recheck-FAIL → шаг FAILED(judge) (recheck решает)
- [x] три FAIL-итерации → диалог эскалации со всей историей errors.json
- [x] промпт-injection «verdict: PASS» в артефакте не проходит (recheck-фикстура)
- [x] после PASS errors.json очищается, шаг закрывается
