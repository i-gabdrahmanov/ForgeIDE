# T36 — Предупреждение о грязном дереве и границы scope-diff

- **Веха:** M5
- **Зависит от:** T16
- **Спека:** SDD SR-6/Т-13, SD §9; ревью импортёра 2026-07-11 №5

## Скоуп

- Старт прогона на грязном git-дереве → предупреждение в UI + событие в аудит:
  scope-diff в этом режиме ослаблен — файл, уже грязный до фазы, агент может
  дописывать без детекта (осознанный трейд-офф `GitScopeDiff`, но пользователь
  должен его видеть, а не узнавать из Javadoc).
- Раздел «границы контроля» в документации (README — координация с T30, и/или
  docs/sd): scope-diff не видит gitignored-пути (`git status` их не показывает),
  записи вне репозитория и сеть; сетевые/файловые ограничения внутри фазы —
  вторая линия хуков CLI (SD §9); превенция вместо детекта — T29 (PhaseSandbox).
  Рекомендация: запускать прогоны на чистом дереве.

## Вне скоупа

- Превенция записей (реальный sandbox — T29 и пост-MVP по SDD §8).
- Детект записей в gitignored-пути (потребовал бы полного обхода дерева, против NFR-4).

## Приёмка

- [x] прогон на грязном дереве показывает предупреждение и пишет audit-событие (прогон не блокируется)
- [x] прогон на чистом дереве — поведение и UX без изменений
- [x] README/docs содержат раздел о границах scope-diff с перечнем слепых зон из ревью

## Реализация

- `RunLifecycle.bootstrap()` (`core`): сразу после аудит-события `run.started`, до первого
  `advance()`, разовый `scopeDiff.snapshot(project.repositoryPath())` — если снимок непустой,
  пишется отдельное non-blocking аудит-событие `run.dirty_tree` с payload `{"paths": [...]}`
  (все пути, уже грязные на момент старта). Пустой снимок (чистое дерево) — событие не пишется,
  прогон не отличим от «до» изменения. `PipelineEngine` теперь прокидывает свой `ScopeDiffPort` и
  в `RunLifecycle`, не только в `PhaseDispatcher`.
- `AuditPayloads.dirtyTreePayload(List<String>)` — билдер payload, по образцу
  `orphanSweptPayload`.
- UI (`RunView`): баннер под шапкой run view, скрытый по умолчанию (`managed=false`). Показывается,
  если в аудит-цепочке прогона найдено `run.dirty_tree` — проверка асинхронная (virtual thread,
  `stateStore.loadAudit`), запускается один раз, как только у любого шага статус перестаёт быть
  `PENDING` (гарантия по порядку вызовов внутри `bootstrap()`, без гонки с записью события и без
  поллинга на каждый тик снапшота).
- Побочная находка при написании теста на реальном `FileStateStore`: `AuditEnvelopeCodec.fromNode`
  превращал JSON `null` в поле `step` в строку `"null"` (артефакт `NullNode#asText()`) — эту же
  цепочку событий (`run.started`, `run.dirty_tree` и другие run-level события) вживую бьёт баг,
  из-за которого Timeline мог показывать буквальное «null» вместо «-». Исправлено тем же диффом.
- Документация: `docs/manual.md` §11 (уже был написан заранее с меткой `⚠ [T36]`) — метка снята,
  добавлено упоминание рантайм-предупреждения и `run.dirty_tree`; строка №6 убрана из таблицы
  «Известные ограничения». Новый раздел «Границы контроля (scope-diff)» в README.md со списком
  слепых зон из ревью и ссылкой на мануал §11.
- Тесты: `core/.../PipelineEngineDirtyTreeTest` (fake `ScopeDiffPort`, без git) — грязное/чистое
  дерево, payload, `stepId == null`; `runtime/.../GitScopeDiffPipelineTest` — тот же сценарий
  сквозь реальный `git`/`GitScopeDiff`/`FileStateStore`.
