# БТ — ForgeIDE: IDE для составления обвязок (harness) агентных систем

> Целевые рантаймы: claude (Claude Code), qwen, gemini, gigacode (форк).
> Референс-пайплайн: Forge (`~/Documents/dev/skills/forge` — feature-pipeline / forgelite, PDLC v3.5).

---

## 1. Проблема и мотивация

Сегодня оркестратором пайплайна выступает **сам агент** (скилл `feature-pipeline` грузится в контекст
модели, и модель «ведёт» фазы). Скиллы — это guidance, модель может их проигнорировать. Практика
прогонов (см. `forge/issues.md`) это подтверждает:

- модель пропускает фазы (TDD, tech-design) или выполняет их inline вместо субагента;
- перестаёт задавать вопросы человеку — гейты «проскакивают», диалоги не появляются;
- пишет мусор в артефакты и сама же прогоняет судью, который говорит «норм»;
- хуки (enforcement) — последняя линия обороны, но они fail-open по таймауту и живут
  внутри процесса агента: модель-оркестратор ищет пути обхода.

**Корень проблемы: state machine пайплайна живёт в контексте модели.** Всё, что форсится текстом,
обходимо.

**Решение: вынести оркестрацию из модели в детерминированное Java-приложение.**
Агент понижается до **исполнителя одной фазы** (headless-вызов), а последовательность фаз, гейты,
судьи, стейт и лимиты контролирует код, до которого модель дотянуться не может.

Принцип PDLC сохраняется и усиливается: `Pipeline > model` — теперь буквально,
пайплайн исполняется вне модели.

---

## 2. Концепция

**ForgeIDE = десктоп-IDE + детерминированный оркестратор (control-plane) в одном приложении.**

| Слой | Кто исполняет | Что делает |
|---|---|---|
| Оркестрация | **Java-приложение** | граф шагов, переходы, ретраи, лимиты итераций, стейт |
| Гейты человека | **Java UI** | нативные диалоги подтверждения — модель не может их «не показать» |
| Судьи / execution-gates | **Java-приложение** | запускает `check_*.py` / `run_judge.py` как подпроцессы, вердикт хранит у себя |
| Исполнение фазы | **агент (headless)** | `claude -p` / `gigacode --experimental-hooks -p` / qwen — один вызов = одна фаза |
| Скрипты | **Java-приложение** | python-шаги пайплайна как подпроцессы |
| Хуки | внутри процесса агента | остаются как **вторая линия обороны** (defense-in-depth) |

Ключевое отличие от «агент-оркестратора»: агент **не видит манифест, не закрывает шаги,
не решает, что дальше**. Он получает промпт фазы + контекст, возвращает артефакт + JSON-результат.
Всё остальное — детерминированный код.

## 3. Пользователь и сценарии

Пользователь — инженер, внедряющий агентный конвейер на проекте (Java/Spring и далее — любой стек).

1. **Собрать обвязку**: создать проект, накидать пайплайн из плиток, отредактировать промпты, сохранить.
2. **Прогнать фичу**: запустить пайплайн, отвечать на гейтах, смотреть живой лог, получить PR.
3. **Дорабатывать харнес**: по итогам прогона поправить промпт плитки / скрипт судьи, перезапустить с места падения.

## 4. Функциональные требования

### 4.1 Проект

- Пользователь может создать проект и положить в него: адрес репозитория, спецификации,
  доп. параметры. Форма расширяемая (произвольные key-value / JSON-параметры, доступные плиткам
  как переменные `${...}`).
- Проект хранит: подключённые рантаймы (claude/qwen/gemini/gigacode + путь к бинарю и флаги,
  напр. `--experimental-hooks`), профиль критичности (low/medium/high → auto_max_risk R2/R1/R0),
  токен-бюджет.
- **Импорт существующей обвязки**: указать каталог Forge-репо (`hooks/` + `skills/`) — IDE
  разбирает SKILL.md / settings.hooks.json и строит из них пайплайн-граф. Это MVP-путь: не рисовать
  feature-pipeline с нуля, а импортировать готовый.

### 4.2 Конструктор пайплайна (canvas)

- Пользователь собирает пайплайн из **плиток** на канвасе, связи = зависимости (`depends_on`),
  как в манифесте pipeline-state.
- Типы плиток:
  - **Агент-шаг** — промпт фазы (контракт субагента), рантайм, ожидаемые артефакты, `step_id`;
  - **Скрипт** — python/shell-шаг (`prepare_design_context.py`, `add_steps.py`, ...);
  - **Судья** — LLM-судья и/или детерминированный `check_*.py`; политика FAIL (лимит итераций,
    возврат на шаг N, эскалация человеку);
  - **Гейт человека** — текст вопроса, варианты ответа, что показать (диff, артефакт);
  - **Условие/ветка** — маршрутизация (full/lite — как router, но выбирает человек в UI, а не модель);
  - **Цикл per-task** — размножение подграфа по элементам task-plan.json (04-test/04-build/07-deliver на задачу).
- Валидация графа при сохранении: циклы, недостижимые шаги, гейт без судьи перед деливери и т.п.
- Формат хранения пайплайна — **декларативный файл** (YAML/JSON) в репо проекта: версионируется,
  диффается, ревьюится. Канвас — редактор этого файла, не замена ему.

### 4.3 Режим редактирования

- По плитке видно и можно редактировать: промпт агента / SKILL.md, скрипт, конфиг судьи.
- Редактор с подсветкой (markdown, python, json), diff с последней версией.
- Библиотека плиток: реестр скиллов (аналог SKILLS-REGISTRY.md — owner / validity / scope /
  eval coverage) — плитка «протухла» → подсветка в UI.

### 4.4 Режим прогона

- Запуск пайплайна по фиче (slug, Jira-ключ). На канвасе — живой статус шагов
  (pending / running / passed / failed / waiting-gate).
- **Гейт = модальный диалог IDE** с контекстом (артефакт, дифф, вердикт судьи). Ответы пишутся
  в стейт с указанием, кто и когда подтвердил.
- Живой лог: tail JSONL-аудита + вывод текущего агента (stream-json), фильтр по шагу.
- FAIL судьи → автоперезапуск фазы с накопленными ошибками (errors.json), максимум N итераций
  (лимит форсится оркестратором, дефолт 3) → эскалация человеку: сброс / отмена / override.
- Override — только через UI, с обязательной причиной, фиксируется в аудите (аналог `override_judge.py`).
- **Резюмируемость**: прогон можно остановить и продолжить с последнего закрытого шага;
  обрыв стрима агента = ретрай шага, а не потеря пайплайна.
- Параллельные фичи: несколько прогонов одновременно (namespace по фиче, как в pipeline-state).

### 4.5 Стейт, аудит, наблюдаемость

- Манифест шагов ведёт **только IDE** (файл/встроенная БД). Агенту он доступен read-only
  (выжимкой в промпт), запись — никогда. `state-write-guard` из хука превращается в
  структурную гарантию.
- Append-only журнал аудита: каждый вызов агента (промпт, рантайм, токены, длительность,
  результат), каждый вердикт судьи, каждый ответ на гейте.
- Дашборд метрик по прогонам: токены/фаза, итерации судей, длительность, стоимость
  (аналог `agentops.py`).
- Архив логов агентов по прогонам (`ai-logs/<feature>/iter-NN/`).

## 5. Инварианты безопасности (зачем Java-оркестратор)

Это ядро ценности продукта — то, что «агент-оркестратор» гарантировать не может:

1. **Модель не управляет переходами.** Закрыть шаг / пропустить фазу / переставить порядок — только код IDE.
2. **Гейт нельзя не показать.** Диалог человеку — вызов UI, а не `ask_user_question`, который модель «забывает».
3. **Судья запускается оркестратором,** вердикт хранится вне досягаемости агента. «Сам прогнал судью — сам сказал норм» невозможно.
4. **Лимиты — в рантайме:** итерации на фазу, таймаут процесса, токен-бюджет (превышение = kill процесса, не «warn в контекст»).
5. **Изоляция фаз структурная:** одна фаза = один headless-процесс со своим контекстом. Inline-исполнение субагентной фазы исключено конструкцией, а не хуком.
6. **Хуки остаются** внутри агент-процессов как вторая линия (risk ladder, destructive-blocker, pii-boundary) — defense-in-depth, а не единственная защита.
7. **Полный аудит** без участия модели: журнал пишет оркестратор.

## 6. Интеграция с рантаймами агентов

- Абстракция `AgentRuntime`: адаптеры для claude / qwen / gemini / gigacode.
  Общий контракт: запуск headless (`-p`), передача промпта + рабочей директории,
  чтение потокового JSON-вывода, код возврата, извлечение итогового JSON (`step_id`, артефакты, `pending_questions`).
- Вопросы субагента — через `pending_questions` в результате: IDE показывает их человеку
  и перезапускает шаг с ответами (паттерн уже отработан в jira-task-writer).
- Деплой хуков в целевой проект (`<project>/.gigacode/` / `.claude/`) — кнопкой из IDE
  (обёртка над deploy.sh) + preflight-проверка «харнес активен» перед прогоном.
- MCP-интеграции (Jira, Bitbucket) остаются на стороне агент-фаз; IDE передаёт креды/конфиг.

## 7. Технологический стек (прототип)

- **Java 21**, сборка Gradle.
- **UI: JavaFX** — рекомендация для прототипа: зрелый десктоп-фреймворк, канвас/графы делаются
  нативно, редактор кода — RichTextFX, диаграммы — свои узлы на Canvas/Scene Graph.
  Альтернативы, если JavaFX не зайдёт: Swing + FlatLaf (консервативно), Compose Multiplatform
  (Kotlin), встроенный web-UI (Spring Boot + браузер) — последнее ломает «десктопность», не рекомендуем.
- Процессы — `ProcessBuilder` + виртуальные потоки (Java 21) для параллельных шагов и tail-логов.
- Стейт/аудит — файлы (JSON/JSONL, совместимо с текущим pipeline-state) либо встроенная
  SQLite/H2 для индексов и метрик; артефакты — всегда файлы в репо проекта.
- Пайплайн-описание — YAML/JSON-схема с валидацией.

## 8. Prior art (что уже есть в интернете и чем мы отличаемся)

- **Node-based конструкторы LLM-пайплайнов**: [Langflow](https://blog.elest.io/dify-vs-langflow-vs-flowise-which-open-source-llm-app-builder-actually-ships-to-production/),
  [Flowise, Dify](https://toolhalla.ai/blog/dify-vs-flowise-vs-langflow-2026) — подтверждают UX-паттерн
  «плитки + канвас + инспектор», но это web/python-платформы для чат-ботов/RAG, а не обвязка
  кодинг-агентов с гейтами и TDD.
- **Детерминированные оркестраторы агентов**: [Microsoft Conductor](https://opensource.microsoft.com/blog/2026/05/14/conductor-deterministic-orchestration-for-multi-agent-ai-workflows/)
  (YAML-workflow, детерминированный роутинг между агентами), Bernstein (параллельные Claude
  Code/Codex/Gemini, координация без LLM-токенов) — подтверждают сам подход «оркестрация кодом,
  агент = исполнитель», но это CLI без IDE и без редактирования харнеса.
- **Headless-режимы рантаймов** ([Claude Code headless](https://www.mindstudio.ai/blog/claude-code-headless-mode-autonomous-agents),
  qwen/gigacode `--experimental-hooks -p`) — штатный механизм интеграции, на котором строим адаптеры.

**Ниша ForgeIDE**: десктоп-IDE, где в одном инструменте (а) визуальное редактирование харнеса
(промпты, судьи, хуки) и (б) детерминированный прогон с гейтами вне досягаемости модели. Такой
связки в готовом виде нет.

## 9. Этапы

**MVP (прототип):**
1. Проект: создание, репо, параметры, подключение рантайма (один — gigacode или claude).
2. Импорт Forge-обвязки → пайплайн-граф на канвасе (read-only достаточно для старта).
3. Прогон **forgelite** (короткая ветка: jira → ground → design → RED → GREEN → verify → deliver)
   с гейтами-диалогами, живым логом и стейтом в IDE.
4. Режим редактирования промптов плиток.

**Дальше:**
- полный feature-pipeline (BRD → … → deliver, судьи, eval-plan, per-task циклы);
- мульти-рантайм и сравнение прогонов (одна фича на claude vs qwen);
- редактор плиток с нуля + библиотека/реестр (validity, eval coverage);
- дашборд метрик и стоимости; экспорт пайплайна в CI (headless-режим самой IDE).

## 10. Открытые вопросы

1. Формат пайплайн-файла: свой YAML или совместимость с манифестом pipeline-state как он есть?
2. Гейты удалённо: прототип — только локальный десктоп, или сразу заложить возможность ответить на гейт с телефона/веба?
3. Судьи-LLM: гонять через тот же рантайм агента или отдельным лёгким вызовом API?
4. Лицензии/приватность: логи агентов содержат код проекта — политика хранения и маскирования (PII-boundary на стороне IDE).
