# SDD — ForgeIDE: спецификация системы

> Основание: [БТ](../bt/init-bt.md) → [SD](../sd/system-design.md) → этот документ.
> Формат сценариев: GWT (Дано / Когда / Тогда). Требования: **FR-x** (функциональные),
> **SR-x** (безопасность, выведены из модели угроз §4), **NFR-x** (нефункциональные).

---

## 1. Термины

| Термин | Значение |
|---|---|
| Движок | детерминированный оркестратор внутри IDE (актор, один поток мутаций) |
| Агент-фаза | один headless-вызов рантайма (claude/qwen/gigacode) = один шаг пайплайна |
| Проекция манифеста | копия стейта в формате pipeline-state Forge в `<project>/ground/…` для хуков |
| Source of truth (SoT) | стейт прогона в `~/.forgeide/state/…`, пишет только движок |
| Гейт | точка подтверждения человеком, модальный диалог IDE |
| Судья | проверка результата шага: детерминированный скрипт и/или LLM-вызов |
| Outward-действие | действие с внешним эффектом: `git push`, создание PR, запись в Jira |
| Trusted / untrusted | артефакт движка / любой вывод модели |

---

## 2. Акторы и границы доверия

| Актор | Доверие | Возможности |
|---|---|---|
| Человек-оператор | trusted | гейты, override, редактирование обвязки |
| Движок IDE | trusted | переходы, судьи, outward-действия, аудит |
| Агент-фаза (модель) | **untrusted** | shell/Write/Edit в рабочей директории **с правами пользователя ОС** |
| Хуки в агент-процессе | trusted-код в untrusted-среде | вторая линия, fail-open по таймауту |
| Скрипты судей | trusted при целостности | детерминированные вердикты |

**Ключевой факт границы доверия:** агент-процесс работает под тем же пользователем ОС, что и IDE.
Физически он МОЖЕТ писать в `~/.forgeide`, править хуки, скрипты судей и pipeline.yaml. Поэтому
гарантии делятся на два класса, и спецификация всюду указывает класс:

- **P (prevent)** — обход структурно невозможен (нет пути кода);
- **D (detect)** — обход возможен, но детерминированно обнаруживается до того, как повлияет
  на переходы/доставку, шаг фейлится, инцидент в аудите.

## 3. Функциональные требования

### FR-1 Проект

- **FR-1.1** Создание проекта: имя, путь/URL репозитория, спецификации, произвольные params
  (key-value, описываются в `pipeline.yaml → params`, форма генерируется динамически).
- **FR-1.2** Подключение рантаймов: путь к бинарю, флаги, проверка доступности (`--version`).
- **FR-1.3** Профиль критичности low/medium/high → `auto_max_risk` R2/R1/R0 (совместимо с risk-policy Forge).
- **FR-1.4** Деплой обвязки в проект кнопкой (обёртка `deploy.sh`) + preflight.

```gwt
Дано: проект без задеплоенной обвязки
Когда: пользователь нажимает «Запустить прогон»
Тогда: движок запускает preflight; exit≠0 → прогон НЕ стартует,
       показано «enforcement off» с диагностикой preflight
```

### FR-2 Пайплайн: файл и канвас

- **FR-2.1** Пайплайн хранится в `<project>/.forgeide/pipeline.yaml`; канвас — его редактор.
- **FR-2.2** Типы шагов: `agent`, `script`, `judge`, `gate`, `branch`, `per_task_loop`, `outward` (контракты — §5.1).
- **FR-2.3** Валидация при загрузке/сохранении: ацикличность, достижимость, существование ссылок
  `target`/`routes`/`prompt`, наличие судьи перед каждым outward-шагом. Ошибка → координата шага + подсветка.
- **FR-2.4** Плитка показывает: промпт/скрипт/конфиг, статус validity из реестра (протухла → warning).

**Конструктор своих пайплайнов:**

- **FR-2.5** Палитра типов плиток; перетаскивание на канвас создаёт шаг с уникальным id;
  протяжка ребра между плитками = `depends_on`; удаление/дублирование плитки; undo/redo.
- **FR-2.6** Live-валидация: тот же валидатор, что при загрузке, гоняется на каждое изменение
  графа; ошибки — бейджами на плитках, запуск прогона заблокирован до зелёной валидации.
- **FR-2.7** Канвас и YAML-редактор — два представления одной модели (вкладки, переключение
  без потери правок). Сохранение с канваса регенерирует YAML; комментарии в YAML при этом
  не сохраняются — фиксируем как осознанное ограничение; перед сохранением показывается diff.
- **FR-2.8** Создание agent-плитки «с нуля»: IDE генерирует prompt-файл из шаблона с обязательным
  контрактом (§5.2: итоговый JSON, `step_id`, запрет самостоятельных outward-действий) — пользователь
  дописывает содержание, контрактная часть подсвечена и защищена от случайного удаления (predупреждение).
- **FR-2.9** Библиотека плиток: проектная (`.forgeide/library/`) и пользовательская
  (`~/.forgeide/library/`); плитку/подграф можно сохранить в библиотеку и вставить в другой
  пайплайн (промпт+конфиг копируются, пути перевязываются).

```gwt
Дано: пустой канвас нового пайплайна
Когда: пользователь перетаскивает agent-плитку, judge-плитку, соединяет их,
       но не заполняет prompt у agent-шага
Тогда: на agent-плитке бейдж «нет промпта», кнопка запуска неактивна
       с перечнем ошибок валидации; после заполнения — валидация зелёная
```

### FR-3 Прогон (движок)

- **FR-3.1** Запуск по фиче (slug + params); параллельные прогоны разных фич независимы.
- **FR-3.2** Шаг готов, когда все `depends_on` в `PASSED`; диспатч на виртуальном потоке;
  все мутации стейта — только в потоке-акторе.
- **FR-3.3** Каждый переход персистируется в SoT ДО публикации события наружу.
- **FR-3.4** Резюмируемость: рестарт IDE → восстановление из SoT; шаги `RUNNING` → `FAILED(interrupted)` → предложение ретрая.

```gwt
Дано: прогон с шагом lite-green в RUNNING; IDE убита (kill -9)
Когда: IDE запущена снова и прогон открыт
Тогда: все PASSED-шаги сохранены; lite-green в FAILED(interrupted);
       доступна кнопка «повторить шаг»; аудит содержит событие recovery
```

- **FR-3.5** Снапшот определения: `pipeline.yaml` + все промпты читаются ОДИН раз на старте прогона
  в память/SoT; изменения файлов на диске не влияют на идущий прогон (см. Т-13).

### FR-4 Агент-фазы

- **FR-4.1** Контракт вызова: промпт через stdin; stdout = stream-json; stderr отдельно (SD §6.1).
- **FR-4.2** Контракт результата — итоговый JSON (§5.2); отсутствие `result`-события = `FAILED(stream)`.
- **FR-4.3** Валидация артефактов: каждый путь из `expects` существует, непустой, соответствует
  типу (md/json парсится) — иначе `FAILED(artifacts)` без участия судьи.
- **FR-4.4** `pending_questions`: полный цикл — FR-10 (лимиты — против Т-15).
- **FR-4.5** Ре-итерация по FAIL судьи: ≤ N (дефолт 3) с накоплением `accumulated_errors`; далее
  эскалация: сброс / отмена / override (причина обязательна, в аудит).

### FR-5 Гейты

- **FR-5.1** Гейт = модальный диалог: вопрос, варианты, артефакты/диффы. Ответ + пользователь ОС +
  timestamp → SoT и аудит.
- **FR-5.2** Гейт показывает **реальные данные** (файл с диска, `git diff`), а не пересказ модели;
  текст модели в UI маркируется как untrusted (см. Т-5).
- **FR-5.3** Для шагов с риском R2+ (профиль критичности) — подтверждение с явным чекбоксом
  «я посмотрел дифф» (см. Т-17).

```gwt
Дано: шаг gate-deliver в WAITING_GATE
Когда: пользователь подтверждает
Тогда: outward-действия выполняет ДВИЖОК (git push, создание PR) — не агент;
       результат и ссылки в аудит; отказ → ветка отклонения по routes
```

### FR-6 Судьи

- **FR-6.1** Судья = композиция: опциональный LLM-вердикт + ОБЯЗАТЕЛЬНЫЙ детерминированный
  recheck для судей, блокирующих outward-шаги (см. Т-8).
- **FR-6.2** LLM-судья запускается движком отдельным вызовом; verdict.json читает движок;
  вердикты хранятся в SoT, агенту недоступны.
- **FR-6.3** Скрипты судей исполняются из **IDE-кэша обвязки** (см. SR-7), не из рабочей копии проекта.

### FR-7 Стейт, аудит, наблюдаемость

- **FR-7.1** SoT: `~/.forgeide/state/<project-hash>/<pipeline>/<feature>/run.json`.
- **FR-7.2** Проекция манифеста в формате pipeline-state перед каждой агент-фазой; tamper-hash после (SR-2).
- **FR-7.3** Аудит — hash-chain JSONL (§5.3): каждая запись содержит hash предыдущей; разрыв цепи
  детектируется при загрузке (см. Т-2).
- **FR-7.4** Логи фаз: `stdout.jsonl` / `stderr.log` / `meta.json` (SD §6.2); совместимость с `watch-agents.sh`.
- **FR-7.5** Метрики прогона: токены/фаза, итерации, длительность — по аудиту.
- **FR-7.6** Timeline прогона: хронология событий аудита с фильтрами (шаг, тип события,
  «только инциденты»); клик по событию → payload + ссылки на raw-логи и meta.json итерации.
- **FR-7.7** Вкладки лога шага: parsed-события (tool_use/usage/result) / raw stdout / stderr /
  meta; полнотекстовый поиск; follow-режим для идущей фазы.
- **FR-7.8** Экспорт прогона одним архивом (audit.jsonl + meta + raw-логи + вердикты) —
  для расследования инцидента или багрепорта.
- **FR-7.9** История прогонов фичи: список с итогами (статусы шагов, итерации, токены,
  overrides); сравнение двух прогонов — пост-MVP.

### FR-8 Редактирование (промпты, скрипты судей, обвязка)

- **FR-8.1** Редактор промпта/скрипта/конфига плитки с подсветкой и diff к git-версии.
- **FR-8.2** Правка промпта при идущем прогоне действует только со СЛЕДУЮЩЕГО запуска шага
  и фиксируется в аудите (кто, когда, diff-hash). Итерация после правки помечается в timeline.
- **FR-8.3** Редактирование скриптов судей и хуков — **через IDE как trusted-путь**: сохранение
  из редактора атомарно обновляет (1) рабочую копию в проекте, (2) IDE-кэш обвязки (SR-7),
  (3) hash-манифест обвязки (SR-8) + событие аудита `harness.edited` с диффом. Так разрешается
  конфликт с SR-8: правка через IDE легитимна и пересчитывает эталон; правка мимо IDE
  (агентом или внешним редактором) остаётся дрейфом → `STOPPED(harness-drift)` с диффом,
  человек может принять изменения (пересчёт эталона с аудитом) или откатить.
- **FR-8.4** Dry-run судьи: кнопка «прогнать судью» исполняет скрипт на артефактах последней
  итерации целевого шага БЕЗ влияния на стейт — вердикт показывается, не сохраняется.
  Цикл «правка check-скрипта → dry-run → сохранить» не требует перезапуска фаз.
- **FR-8.5** Предпросмотр рендера промпта: подстановки `${...}` + блоки `accumulated_errors`/
  `answers` в том виде, в каком фаза получит их при следующем запуске.

```gwt
Дано: скрипт check_tests_red.py открыт в редакторе IDE, идёт прогон
Когда: пользователь сохраняет правку
Тогда: рабочая копия, кэш SR-7 и hash-манифест SR-8 обновлены атомарно,
       в аудите harness.edited с диффом; следующая агент-фаза проходит проверку hash

Дано: тот же файл правит агент внутри фазы
Когда: движок готовит следующую фазу
Тогда: STOPPED(harness-drift) — дифф показан человеку: принять или откатить
```

### FR-9 Импорт обвязки

- **FR-9.1** Шаблоны `forgelite` / `feature-pipeline` в ресурсах IDE; импортёр привязывает пути
  (промпты из `subagent-prompts.md` по §-заголовкам, `check_*.py`, hooks) к плиткам.
- **FR-9.2** Несматченные плитки подсвечены, привязка вручную; реестр из `SKILLS-REGISTRY.md`.

### FR-10 Вопросы от модели (`pending_questions`)

- **FR-10.1** Контракт (§5.2): `{id, text, type: choice|text, options?, context?}` —
  `context` = путь к артефакту/фрагменту, который показывается рядом с вопросом.
- **FR-10.2** Непустой `pending_questions` в результате → шаг переходит в **`WAITING_INPUT`**
  (персистится в SoT, переживает рестарт IDE). Ветка прогона стоит, независимые ветки продолжаются.
- **FR-10.3** UI — та же инфраструктура, что у гейтов: форма (radio для `choice`, textarea для
  `text`), рядом рендер context-артефакта. Отвечать можно не сразу: диалог закрывается,
  на плитке остаётся бейдж «N вопросов», диалог открывается повторно с канваса.
- **FR-10.4** Ответы → перезапуск фазы: тот же промпт + канонический блок `answers`
  (`id → ответ`); полный отрендеренный текст — в meta.json новой итерации. Раунд вопросов
  НЕ расходует счётчик судейских ре-итераций (FR-4.5).
- **FR-10.5** Лимит: ≤ 2 раундов вопросов на одну итерацию фазы; третий → эскалация:
  «фаза не может завершиться без диалога» с действиями: открыть промпт в редакторе (FR-8) /
  разбить шаг / отменить.
- **FR-10.6** Аудит: `question.asked` (список целиком) и `question.answered` (кто, когда, ответы).

```gwt
Дано: фаза jira вернула result с pending_questions [выбор Epic, номер спринта]
Когда: движок обрабатывает результат
Тогда: шаг → WAITING_INPUT, бейдж «2 вопроса» на плитке; после ответов фаза
       перезапущена с блоком answers; аудит содержит question.asked и question.answered
```

### FR-11 Обработка ошибок

- **FR-11.1** Каждый сбой шага типизирован (таблица ниже). Движок не «глотает» ошибки:
  любой сбой = терминальный статус + событие аудита + доступное действие в UI.
- **FR-11.2** Политика ретраев per-step в pipeline.yaml: `retry: {stream: 1, script: 0}`
  (дефолты глобальные). Авторетрай только для классов, где он безопасен (stream, script);
  инциденты безопасности автоматически не ретраятся никогда.
- **FR-11.3** Диалог эскалации (лимит итераций судьи или инцидент): история ошибок по итерациям
  (errors.json), diff артефактов между итерациями, действия — «повторить» / «повторить с правкой
  промпта» (открывает FR-8) / «сбросить цепочку шагов» / «отменить прогон» / «override» (причина
  обязательна).
- **FR-11.4** Сбой самой IDE (исключение движка): прогон → `PAUSED(engine-error)`, стектрейс в лог
  IDE; стейт не повреждается (атомарная запись SR-3), после рестарта — штатный recovery (FR-3.4).

| Код | Причина | Автодействие движка | Действия человека |
|---|---|---|---|
| `FAILED(stream)` | EOF без result-события / непарсящийся итог | ретрай по политике (дефолт ×1) | повторить / отменить |
| `FAILED(artifacts)` | expects отсутствуют / пустые / не парсятся | ре-итерация с errors (≤N) | как FAIL судьи |
| `FAILED(judge)` | детерминированный recheck FAIL | ре-итерация с accumulated_errors (≤N) → эскалация | FR-11.3 |
| `FAILED(scope)` | запись вне `allowed_write` (SR-6) | инцидент, без авторетрая | посмотреть дифф, откатить лишнее, повторить |
| `FAILED(tampered)` | tamper-hash проекции (SR-2) | восстановление проекции, инцидент | расследование по timeline, повторить |
| `FAILED(budget)` | токены / wall-clock / кап вывода | kill process group | поднять бюджет шага, повторить |
| `FAILED(interrupted)` | kill IDE / отмена пользователем | recovery при старте | повторить |
| `STOPPED(harness-drift)` | hash обвязки не сходится (SR-8) | стоп прогона целиком | принять дифф (FR-8.3) или откатить |
| `STOPPED(audit-chain)` | разрыв hash-chain аудита (SR-3) | прогон блокируется | экспорт (FR-7.8) на расследование; новый прогон |
| script exit≠0 | сбой script-шага | ретрай по политике | stderr → правка скрипта (FR-8) → повторить |
| gate declined | человек отклонил гейт | ветка `routes` отклонения либо возврат на шаг | — |

## 4. Модель угроз и сценарии обхода

**Нарушитель:** модель внутри агент-фазы. Мотивация не важна (ленивая оптимизация, галлюцинация,
prompt-injection из кода/тикетов проекта) — важно, что она имеет shell с правами пользователя ОС
и историческую склонность «срезать путь» (см. `forge/issues.md`). Второй нарушитель — усталый
человек (гейт-фатиг). IDE и ОС считаются доверенными.

### 4.1 Реестр сценариев обхода

| ID | Сценарий обхода | Вектор | Контрмера | Класс |
|---|---|---|---|---|
| Т-1 | Модель правит проекцию манифеста (закрывает себе шаги) | Write/redirect/`python -c` в `ground/statements/…` | state-write-guard (хук) + SR-2 tamper-hash после каждой фазы: расхождение → `FAILED(tampered)`, проекция перезаписывается из SoT | D |
| Т-2 | Модель находит и правит SoT / аудит в `~/.forgeide` | пути угадываются (конвенция), права ОС общие | SR-1: пути не сообщаются агенту (env/промпт чисты); SR-3: hash-chain аудита + контрольная сумма run.json, проверка при каждой загрузке; разрыв → прогон блокируется, инцидент | D |
| Т-3 | Подделка вердикта судьи (правка verdict.json/errors.json/evals-кэша) | файлы в проекте доступны на запись | вердикты живут в SoT; ингест LLM-вердикта всегда завершается детерминированным recheck, запускаемым движком | P |
| Т-4 | Агент сам делает push/PR/Jira-запись до гейта | shell/MCP внутри фазы | SR-4: outward-действия исполняет ТОЛЬКО движок после гейта; SR-5: креды (git-токены, MCP) не попадают в env фаз, которым не положены; хуки evidence-enforcer/gate-guard — вторая линия | P (по кредам) / D (локальный git) |

### 4.2 Требования безопасности (сводно)

- **SR-1** Ни промпт, ни env, ни working dir агент-фазы не содержат путей `~/.forgeide`.
- **SR-2** После каждой агент-фазы: SHA-256 проекции манифеста сверяется с ожидаемым; расхождение →
  `FAILED(tampered)` + восстановление проекции + инцидент.
- **SR-3** Аудит — hash-chain (`hash = SHA-256(prev_hash + payload)`); `run.json` пишется атомарно
  (tmp + rename) с контрольной суммой; проверка целостности при каждой загрузке прогона.
- **SR-4** Outward-действия (`git push`, PR, Jira-запись) выполняет движок детерминированным кодом
  после соответствующего гейта. Агент-фазы готовят изменения только локально.
- **SR-5** Env-скоупинг: каждый шаг получает env строго по `env_scope` из pipeline.yaml;
  git/MCP-креды хранятся в конфиге IDE и не попадают в фазы без скоупа.
- **SR-6** Scope-diff: после каждой агент-фазы затронутые файлы (`git status --porcelain` +
  снимок незатрекаемых) сверяются с масками `allowed_write` шага; нарушение → `FAILED(scope)`.
- **SR-7** Скрипты судей и preflight исполняются из IDE-кэша обвязки (`~/.forgeide/harness-cache/<hash>/`),
  заполняемого при деплое; рабочая копия в проекте — только для хуков агент-процесса.
- **SR-8** Hash-манифест обвязки в проекте (`hooks/`, `skills/`, `settings.hooks.json`) сверяется
  перед каждой агент-фазой; дрейф → прогон останавливается до передеплоя/подтверждения человеком.
- **SR-9** Фаза стартует в новой process group; завершение = kill группы + descendants + sweep
  осиротевших процессов с cwd в проекте.

```gwt
# приёмочный сценарий Т-1/SR-2
Дано: идёт агент-фаза lite-green
Когда: процесс-имитатор агента дописывает в manifest.json строку "lite-verify: completed"
Тогда: по завершении фазы движок фиксирует расхождение hash,
       шаг → FAILED(tampered), manifest.json перезаписан из SoT,
       в аудите событие incident.tamper с диффом

# приёмочный сценарий Т-4/SR-4
Дано: фаза lite-red с env_scope без git-кредов
Когда: агент выполняет git push
Тогда: push падает (нет кредов) ИЛИ блокируется хуком;
       scope-diff фиксирует попытку; шаг → FAILED(scope)

# приёмочный сценарий Т-7/SR-8
Дано: агент в фазе правит .gigacode/hooks/tdd-guard.py
Когда: движок готовит следующую агент-фазу
Тогда: hash-манифест обвязки не сходится → прогон STOPPED(harness-drift),
       человеку показан дифф изменённых файлов обвязки
```

## 5. Контракты данных

### 5.1 `pipeline.yaml` (дополнение к SD §5 полями безопасности)

```yaml
steps:
  - id: lite-green
    type: agent
    runtime: gigacode
    prompt: prompts/lite-green.md
    expects: [src/main/**]               # валидация артефактов (FR-4.3)
    allowed_write: [src/**, build/**]    # маска scope-diff (SR-6)
    env_scope: []                        # секреты не нужны (SR-5)
    budget: {tokens: 2_000_000, wall_clock: 40m, output_mb: 512}
  - id: deliver
    type: outward                        # исполняет ДВИЖОК (SR-4)
    depends_on: [gate-deliver]
    actions: [git_push, create_pr]       # детерминированные экшены движка
    env_scope: [GIT_TOKEN]
```

Новый тип шага **`outward`** — следствие модели угроз: действия с внешним эффектом вынесены
из агент-фаз в детерминированные экшены движка (git push, PR через API, Jira-запись).

### 5.2 Контракт результата агент-фазы (итоговый JSON в stream)

```json
{
  "step_id": "lite-green",
  "status": "done | blocked",
  "artifacts": ["src/main/java/...", "..."],
  "pending_questions": [{"id": "q1", "text": "...", "type": "choice|text",
                         "options": ["..."], "context": "docs/.../tech-design.md#секция"}],
  "summary": "текст для человека (untrusted)"
}
```

Отсутствие итогового JSON / не-парсинг → `FAILED(stream)`. Поле `status` — информационное:
переход шага определяют судьи и движок, НЕ самооценка модели.

### 5.3 Конверт аудита (hash-chain JSONL)

```json
{"seq": 41, "ts": "2026-07-07T12:00:00Z", "run": "uuid", "step": "lite-green", "iter": 2,
 "type": "step.failed | gate.answered | incident.tamper | judge.verdict | ...",
 "payload": {...}, "prev": "sha256...", "hash": "sha256(prev + canonical(payload))"}
```

### 5.4 `meta.json` фазы

`{command[], env_keys[], prompt (полный), prompt_sha256, started/finished, exit_code, usage, head_before, head_after}`
— поля `head_before/after` служат scope-diff (SR-6) и расследованию инцидентов.

## 6. Нефункциональные требования

- **NFR-1** Пропускная способность лога: pump-потоки держат ≥ 10 МБ/с вывода без блокировки
  дочернего процесса; UI-буфер — кольцевой, 10k строк на шаг.
- **NFR-2** Латентность UI: обновление статусов канваса ≤ 200 мс от события движка.
- **NFR-3** Восстановление после kill IDE: ≤ 5 с до готовности прогона к ретраю.
- **NFR-4** Оверхед контролей на фазу (hash обвязки + scope-diff + tamper-hash) ≤ 2 с
  на репозитории до 100k файлов (git plumbing, не полные обходы).
- **NFR-5** Платформы: macOS + Linux (process group семантика POSIX); Windows — пост-MVP.
- **NFR-6** Совместимость: проекция манифеста проходит контракт-тесты против реальных хуков Forge.

## 7. Критерии приёмки

К вехам SD §11 добавляется **анти-обходная приёмка** (веха M3):

1. Набор «злых» фикстур — скрипт-имитатор агент-фазы, реализующий Т-1, Т-4, Т-7, Т-9, Т-12,
   Т-13, Т-14, Т-15 (пишет в манифест, пушит, правит хуки, оставляет фон, правит pipeline.yaml,
   коммитит вне маски, подсовывает пустые артефакты, зацикливает вопросы).
2. Для каждой фикстуры движок обязан завершить шаг соответствующим `FAILED(*)` /
   `STOPPED(harness-drift)` и записать инцидент в аудит — автотест, не ручная проверка.
3. Позитивный e2e: полный forgelite на реальном проекте, все гейты через UI, доставка — только
   через outward-шаг движка.
4. Целостность: обрыв цепи аудита (ручная порча файла) детектируется при следующем открытии прогона.
5. Таксономия FR-11: каждый код ошибки воспроизводится фикстурой и приводит к задокументированному
   статусу + событию аудита (автотест).
6. Конструктор: пайплайн, собранный с нуля из палитры (agent+judge+gate), проходит валидацию
   и исполняется end-to-end на фикстурном рантайме.
