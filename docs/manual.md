# ForgeIDE — мануал пользователя

Как пользоваться ForgeIDE: от пустого проекта до разобранного инцидента прогона.
Все названия кнопок и вкладок — как в текущем UI. Известные ограничения собраны
в [последнем разделе](#известные-ограничения-и-обходные-пути) и помечены по тексту
значком ⚠ со ссылкой на задачу.

## 1. Понятия за пять минут

- **Проект** — git-репозиторий, над которым работают агенты. IDE хранит список
  проектов у себя; в самом репо появляются `.forgeide/` (pipeline) и `.gigacode/`
  (обвязка).
- **Пайплайн** — граф шагов в `<проект>/.forgeide/pipeline.yaml`. Шаг = плитка
  на канвасе.
- **Типы плиток:** `agent` (фаза LLM-агента в отдельном процессе), `script`
  (детерминированный скрипт), `judge` (проверка результата: скрипт и/или
  LLM-судья), `gate` (вопрос человеку, прогон стоит до ответа), `branch`
  (ветвление по условию), `per_task_loop` (повтор вложенных шагов по задачам),
  `outward` (необратимое внешнее действие: push, PR — только после гейта).
- **Обвязка (харнесс)** — `.gigacode/` внутри проекта: скиллы, хуки,
  judge-скрипты. IDE снимает с неё hash-манифест и следит за целостностью.
- **Прогон (run)** — одно исполнение пайплайна для фичи (`feature slug`).
  Переходами управляет только движок; модель не может пропустить гейт, изменить
  вердикт судьи или «продолжить сама».

## 2. Установка и запуск

```bash
./gradlew :ui:run
```

Нужны JDK 21, git, python3. CLI агента (`claude`/`gigacode`) ставится отдельно
и прописывается в проекте (см. ниже). При старте IDE сама подбирает прогоны,
прерванные убийством процесса (шаг `RUNNING` без живого процесса помечается
терминально — его можно продолжить из Run history).

## 3. Проект

**Список проектов → Add / Edit.** Поля формы:

- **Repository path** (`Browse…`) — корень git-репозитория.
- **Criticality** — критичность проекта; влияет на строгость подтверждения
  гейтов (для рискованных гейтов диалог требует явно отметить, что diff
  просмотрен).
- **Spec files** (`Add spec file…`) — файлы спецификации, которые подставляются
  агентам в контекст.
- **Runtimes** (`Add runtime`) — привязка имён `claude` / `gigacode` к бинарям:
  путь, флаги (например `--experimental-hooks`), кнопка **Check availability**
  проверяет, что бинарь запускается. Шаг пайплайна ссылается на рантайм по имени
  (`runtime: gigacode`).
- **Params** — значения параметров пайплайна (`Reload params from pipeline.yaml`
  подтягивает объявленные в YAML). Обязательный параметр без значения блокирует
  запуск — карточка проекта предупредит заранее.

Карточка проекта: **Open canvas** (конструктор), **Start run**, **Run history**,
**Import scaffold** (импортёр обвязки).

## 4. Импорт Forge-обвязки

**Карточка проекта → Import scaffold.** Импортёр не парсит прозу SKILL.md —
он привязывает файлы реальной обвязки к плиткам готового шаблона.

1. Выберите **Шаблон:** `forgelite` (ground → design → gate → red → judge →
   green → judge → gate → deliver) или `feature-pipeline` (BRD → grounding →
   tech-design → Jira → per-task TDD-цикл → доставка).
2. **Выбрать каталог обвязки…** — корень forge-репозитория. Сканер найдёт
   `skills/*/SKILL.md`, `subagent-prompts.md` (контракты режутся на секции по
   заголовкам), `hooks/settings.hooks.json`, скрипты `check_*.py`,
   `SKILLS-REGISTRY.md`.
3. Таблица привязок: agent-плитки матчатся на §-секции по вхождению id шага в
   заголовок, judge-скрипты — по имени файла. Несматченные строки подсвечены —
   выделите строку и нажмите **Выбрать файл вручную…** ⚠ ручная привязка берёт
   выбранный файл целиком, выбрать отдельную §-секцию пока нельзя ([T33]).
4. **Импортировать** — активна, когда несматченных не осталось. В проект
   запишутся: `.forgeide/pipeline.yaml`, каталог `prompts/` (нарезанные
   контракты), `.gigacode/skills/<id>/SKILL.md`, hooks, `SKILLS-REGISTRY.md`
   и манифест соответствия плиток реестру.

Из `SKILLS-REGISTRY.md` берутся owner/validity/scope скиллов: плитка с
просроченной validity получает warning-бейдж на канвасе.

⚠ Известные ограничения импорта: копируется только `SKILL.md` скилла, без
`references/` ([T34]); повторный импорт молча перезапишет локальные правки
промптов ([T35]).

## 5. Обвязка: деплой и preflight

Прогон стартует только если обвязка проекта «задеплоена»: снят hash-манифест
(baseline), прошёл `preflight.py` (существует `.gigacode/settings.hooks.json`,
он валидный JSON, все упомянутые в нём скрипты есть на диске). Иначе прогон
сразу встаёт с причиной `HARNESS_PREFLIGHT`.

⚠ **Кнопки «задеплоить обвязку» в UI сейчас нет** — деплой не привязан ни к
импорту, ни к запуску. Обходной путь: откройте канвас, выберите любую плитку с
файлом обвязки (например, judge-скрипт), внесите тривиальную правку и нажмите
**Save** — сохранение файла под `.gigacode/` через редактор фиксирует новый
baseline обвязки, после чего прогон стартует. Проект, импортированный до T32,
может иметь `settings.hooks.json` в `.gigacode/hooks/` — preflight укажет
точный путь для переноса в корень `.gigacode/`.

После деплоя работает контроль дрейфа: изменение любого файла обвязки вне
доверенного редактора IDE останавливает прогон перед следующей агентной фазой
(`harness-drift`) и показывает diff — принять новую версию или откатить решает
человек. Правка через редакторы IDE (см. §7) дрейфом не считается: она сама
становится новым baseline.

## 6. Канвас и конструктор пайплайна

**Open canvas** открывает конструктор с вкладками:

- **Canvas** — граф плиток. Палитра слева: перетащите тип шага на канвас;
  рёбра — перетаскиванием между плитками. **Undo/Redo** поддерживаются.
  Плитка с ошибкой валидации подсвечена, список ошибок — в панели деталей.
- **YAML** — тот же пайплайн текстом: **Preview diff** и **Apply to canvas**
  для правки руками.
- **Library** — библиотека плиток: сохраняйте настроенную плитку и вставляйте
  её в другие пайплайны.

**Save pipeline.yaml** пишет файл через каноничный writer (покажет diff перед
сохранением; комментарии в существующем YAML не сохраняются).

## 7. Редактирование плиток

Клик по плитке → панель деталей с редакторами:

- **Config** — поля шага (expects, allowed_write, env_scope, budget, retry…).
- **Промпт** (agent-плитки) — редактор с подсветкой markdown; **Preview render**
  показывает промпт с подставленными параметрами, **Diff with git HEAD** —
  отличие от закоммиченной версии.
- **Judge-скрипт** — редактор python; **Прогнать судью (dry-run)** исполняет
  проверку немедленно, без прогона — удобно отлаживать `check_*.py`.

Правки файлов под `.gigacode/` через эти редакторы — «доверенный путь»: они
аудируются (`harness.edited`) и обновляют baseline, а не считаются дрейфом.

Важно про запущенный прогон: промпты снапшотятся на старте прогона. Правка
промпта во время прогона повлияет на шаг только через **Retry step** — тогда
шаг перечитает файл.

## 8. pipeline.yaml — краткий справочник

```yaml
version: 1
id: my-pipeline

params:
  feature_name: {required: true, hint: "..."}

steps:
  - id: my-phase
    type: agent
    runtime: gigacode              # имя из Runtimes проекта
    prompt: prompts/my-phase.md    # шаблон промпта, ${param} подставляются
    expects: ["docs/${feature.slug}/out.md"]   # артефакты, без них шаг FAILED
    allowed_write: [docs/**]       # глобы; запись вне — инцидент SCOPE
    env_scope: [JIRA_TOKEN]        # какие секреты получит процесс (только эти)
    budget: {tokens: 2_000_000, wall_clock: 30m, output_mb: 256}
    retry: {stream: 1, script: 0}

  - id: check-phase
    type: judge
    target: my-phase
    check: {command: [python3, .gigacode/skills/x/scripts/check_out.py], timeout: 5m}
    fail_policy: {max_iterations: 3}   # judge FAIL → повтор target, максимум N раз

  - id: approve
    type: gate
    depends_on: [check-phase]
    question: "Утвердить?"
    options: [Утвердить, Отклонить]
    show: ["docs/${feature.slug}/out.md"]
    risk: R1
```

Прочие типы: `script` (детерминированная команда), `branch` (маршруты по
условию), `per_task_loop` (шаблон вложенных шагов, повторяемый по списку задач),
`outward` (внешнее действие; исполняется движком только после явного гейта).
Полная семантика — [SD §5–6](sd/system-design.md).

## 9. Секреты

Секреты живут в `~/.forgeide/secrets.json` (плоский JSON `{"ИМЯ": "значение"}`,
права 600, атомарная запись). UI-редактора пока нет — файл правится руками.
Фаза получает **только** ключи, перечисленные в её `env_scope`; пустой scope —
пустое окружение.

⚠ Маскирование секретов в логах и вердиктах ещё не реализовано ([T27]) — не
давайте фазам секреты, которые нельзя увидеть в `ground/ai-logs/`.

## 10. Запуск прогона

**Start run** → экран запуска: имя фичи (**Feature slug**, например ключ Jira),
проверка readiness (валидный `pipeline.yaml`, заполненные обязательные params)
→ **Start**.

Экран прогона (**Run view**):

- **Timeline** — шаги со статусами и итерациями; фильтр инцидентов показывает
  tamper/scope/drift-события.
- **Step log** — live-хвост stdout/stderr шага (буфер 10k строк), поиск.
- **Inspector** — конфиг и файлы выбранного шага; judge-скрипт можно править
  прямо здесь (см. §7).
- **Run history** — прошлые прогоны фичи, resume прерванного.

Кнопки: **Cancel run**, **Retry step** (повтор упавшего шага; перечитывает
промпт с диска), **Roll back extra changes** (откат файлов, записанных вне
`allowed_write` — с подтверждением списка), **Export…** (выгрузка прогона).

### Гейты

Когда шаг доходит до `gate`, всплывает диалог: вопрос, кнопки-варианты из
`options`, вкладка **git diff** (что изменилось), **History**. Для рискованных
гейтов надо явно отметить, что diff просмотрен. **«Закрыть (решить позже)»**
прячет диалог, прогон продолжает стоять — плитка на канвасе вернёт диалог по
клику. Ответ уходит в аудит вместе с автором.

### Вопросы модели

Если фаза вернула вопросы, шаг встаёт в `WAITING_INPUT` и открывается диалог:
**«Отправить ответы»** — ответы уйдут шагу на следующей итерации, **«Закрыть
(ответить позже)»** — отложить. Третий раунд подряд (лимит FR-10.5 — не более 2)
эскалирует вместо нового вопроса: диалог с вариантами «Открыть промпт в
редакторе», «Разбить шаг», «Отмена». «Открыть промпт в редакторе» закрывает
диалог так же, как «решить позже» (шаг остаётся `WAITING_GATE`, эскалация
переоткрывается кликом по плитке) и переключает Inspector на вкладку **Prompt**
целевой фазы — правка сохраняется обычным trusted-путём (аудит
`prompt.edited`). Рядом — кнопка **«Повторить с новым промптом»**: отменяет
текущую попытку и сразу повторяет шаг, подхватывая только что сохранённый
промпт.

### Судьи

`judge` исполняет детерминированный скрипт и/или LLM-судью. Скрипт запускается
из **кэша IDE**, а не из рабочей копии — фаза не может переписать проверяющий
её скрипт. Вердикт LLM-судьи всегда перепроверяется детерминированным чеком.
FAIL судьи → target-шаг уходит на новую итерацию с накопленными ошибками в
контексте, до `max_iterations`, дальше — эскалация человеку.

## 11. Что контролируется, а что нет

Гарантии движка (см. [SD §9](sd/system-design.md)):

- переходы/гейты/вердикты — вне досягаемости модели; аудит пишет только движок
  (hash-chain, `~/.forgeide`);
- после каждой агентной фазы: scope-diff (`allowed_write`, сдвиг HEAD),
  tamper-check манифеста; перед каждой фазой — drift-check обвязки;
- бюджеты (токены, wall-clock, объём вывода) обрываются в полёте; осиротевшие
  процессы фазы зачищаются.

Слепые зоны (⚠ [T36] — предупреждение и полная документация):

- **грязное git-дерево ослабляет scope-diff** — файл, изменённый до старта фазы,
  агент может дописывать незаметно. Запускайте прогоны на чистом дереве;
- gitignored-пути и записи **вне репозитория** scope-diff не видит;
- **сеть** внутри фазы IDE не ограничивает — это вторая линия (хуки CLI),
  превенция на уровне ОС — [T29].

## 12. Разбор типовых остановок

| Симптом | Причина | Что делать |
|---|---|---|
| Прогон сразу `paused: HARNESS_PREFLIGHT` | обвязка не задеплоена / нет `settings.hooks.json` в корне `.gigacode/` | §5: обходной путь; если файл лежит в `.gigacode/hooks/` — перенесите его в корень по сообщению preflight |
| `STOPPED (harness-drift)` перед фазой | файл обвязки изменён вне IDE (возможно, прошлой фазой) | посмотреть diff; принять (станет baseline) или откатить из кэша |
| Шаг `FAILED (SCOPE)` | запись вне `allowed_write` или сдвиг HEAD | посмотреть список; **Roll back extra changes**; расширить глобы, если легитимно |
| Шаг `FAILED (TAMPERED)` | фаза трогала проекцию манифеста (control-plane) | инцидент; проекция уже восстановлена движком, разобраться в логе фазы |
| Шаг `FAILED (BUDGET)` | превышены токены/время/объём | поднять `budget` шага или сузить промпт |
| Шаг `FAILED (ARTIFACTS)` | не создан файл из `expects` | смотреть Step log; поправить промпт → **Retry step** |
| Судья валит все итерации | реальный дефект или сломанный чек | **Прогнать судью (dry-run)** на текущем дереве, отладить скрипт |
| Прогон завис в `WAITING_GATE`/`WAITING_INPUT` | закрыли диалог «на потом» | клик по плитке на канвасе откроет диалог снова |
| IDE перезапустили посреди прогона | процесс был убит | Run history → выбрать прогон → resume |
