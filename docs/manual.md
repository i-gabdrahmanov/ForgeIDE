# ForgeIDE — мануал пользователя

Как пользоваться ForgeIDE: от пустого проекта до разобранного инцидента прогона.
Все названия кнопок и вкладок — как в текущем UI. Известные ограничения собраны
в [последнем разделе](#известные-ограничения) и помечены по тексту значком ⚠.

Быстрый маршрут для нетерпеливых: §3 (завести проект) → §4 (Import scaffold) →
§5 (Deploy harness — IDE сама предложит) → §10 (Start run).

## 1. Понятия за пять минут

- **Проект** — git-репозиторий, над которым работают агенты. IDE хранит список
  проектов у себя; в самом репо появляются `.forgeide/` (pipeline) и `.gigacode/`
  (обвязка).
- **Пайплайн** — граф шагов в `<проект>/.forgeide/pipeline.yaml`. Шаг = плитка
  на канвасе.
- **Типы плиток:** `agent` (фаза LLM-агента в отдельном процессе), `script`
  (детерминированный скрипт), `judge` (проверка результата: скрипт и/или
  LLM-судья), `gate` (вопрос человеку, прогон стоит до ответа), `branch`
  (ветвление по условию), `per_task_loop` (повтор вложенных шагов по задачам),
  `outward` (необратимое внешнее действие: push, PR — только после гейта).
- **Обвязка (харнесс)** — `.gigacode/` внутри проекта: скиллы, хуки,
  judge-скрипты. IDE снимает с неё hash-манифест и следит за целостностью.
- **Прогон (run)** — одно исполнение пайплайна для фичи (`feature slug`).
  Переходами управляет только движок; модель не может пропустить гейт, изменить
  вердикт судьи или «продолжить сама».

## 2. Установка и запуск

Нужны JDK 21, git, python3 (подробно про сборку и дистрибутивы — [README](../README.md)).

```bash
./gradlew :ui:run                  # запуск из исходников
./gradlew :ui:platformDistZips     # zip-дистрибутивы: mac-aarch64 и linux-x64
```

CLI агента (`claude`/`gigacode`) ставится отдельно и прописывается в проекте
(§3, Runtimes). При старте IDE сама подбирает прогоны, прерванные убийством
процесса: шаг `RUNNING` без живого процесса помечается терминально — прогон
можно продолжить из **Run history**.

## 3. Проект

**Список проектов → Add / Edit.** Поля формы:

- **Repository path** (`Browse…`) — корень git-репозитория.
- **Criticality** — критичность проекта; влияет на строгость подтверждения
  гейтов (для рискованных гейтов диалог требует явно отметить «Я посмотрел
  diff») и на `auto_max_risk` внешних действий.
- **Spec files** (`Add spec file…`) — файлы спецификации, которые подставляются
  агентам в контекст.
- **Runtimes** (`Add runtime`) — привязка имён `claude` / `gigacode` к бинарям:
  путь, флаги (например `--experimental-hooks`), кнопка **Check availability**
  проверяет, что бинарь запускается. Шаг пайплайна ссылается на рантайм по имени
  (`runtime: gigacode`).
- **Params** — значения параметров пайплайна (`Reload params from pipeline.yaml`
  подтягивает объявленные в YAML). Обязательный параметр без значения блокирует
  запуск — карточка проекта предупредит заранее.

Карточка проекта: **Open canvas** (конструктор), **Start run**, **Run history**,
**Import scaffold** (§4) и раздел **Harness** с кнопкой **Deploy harness** (§5).

## 4. Импорт Forge-обвязки

**Карточка проекта → Import scaffold.** Импортёр не парсит прозу SKILL.md —
он привязывает файлы реальной обвязки к плиткам готового шаблона.

### Шаги импорта

1. Выберите **Шаблон:** `forgelite` (ground → design → gate → red → judge →
   green → judge → gate → deliver) или `feature-pipeline` (BRD → grounding →
   tech-design → Jira → per-task TDD-цикл → доставка).
2. **Выбрать каталог обвязки…** — корень forge-репозитория. Сканер найдёт
   `skills/*/SKILL.md`, `subagent-prompts.md` (контракты режутся на §-секции по
   заголовкам), `hooks/settings.hooks.json` со скриптами хуков, скрипты
   `check_*.py`, `SKILLS-REGISTRY.md`.
3. Таблица привязок. Автоматика матчит agent-плитки на §-секции по вхождению id
   шага в заголовок (id сравнивается как отдельное слово: `fp-red` не схватится
   за заголовок про `fp-red-fix`), judge-скрипты — по имени файла. Два цвета
   подсветки:
   - **красная строка** — не сматчилось: выделите её и нажмите
     **Привязать вручную…**. Если выбранный файл — markdown с заголовками,
     откроется диалог «Выберите §-секцию:» — плитке достанется тело выбранной
     секции, а не весь файл; обычный файл берётся целиком;
   - **оранжевая строка** — неоднозначный автоматч (id встречается в нескольких
     заголовках): клик по **Привязать вручную…** покажет диалог «Неоднозначный
     матч — <id>» со списком секций-кандидатов, выберите нужную.
4. **Импортировать** — активна, когда не осталось ни несматченных, ни
   неоднозначных строк.

### Что попадает в проект

- `.forgeide/pipeline.yaml` — пайплайн по шаблону;
- `prompts/*.md` — нарезанные контракты фаз;
- `.gigacode/skills/<id>/` — **каталог скилла целиком** (SKILL.md,
  `references/`, `scripts/`…), поэтому ссылки скилла на `references/…`
  работают в целевом проекте; мусор (`.git`, `.venv`, `__pycache__`,
  `node_modules`, `build`…) отфильтровывается. Скилл больше 10 МБ вызовет
  предупреждение «Подозрительно большой скилл» — проверьте, не захватили ли
  бинарники;
- `.gigacode/settings.hooks.json` — конфиг хуков, в корень харнесса;
- `.gigacode/hooks/` — сами скрипты хуков (без них preflight не пройдёт);
- `.gigacode/SKILLS-REGISTRY.md` и `.forgeide/import-manifest.json` — реестр и
  манифест соответствия плиток реестру;
- строка `ground/ai-logs/` дописывается в `.gitignore` проекта (сырые логи
  агентов не должны попадать в git — см. §9).

Из `SKILLS-REGISTRY.md` берутся owner/validity/scope скиллов: плитка с
просроченной validity получает warning-бейдж на канвасе.

### Повторный импорт — безопасен

Перед записью импортёр сравнивает каждый файл с тем, что уже лежит в проекте:

- не изменившиеся файлы не перезаписываются вовсе;
- файлы, правленные локально после прошлого импорта (например, промпты через
  инспектор), показываются в диалоге **«Конфликты при повторном импорте»** —
  выбирайте **«Перезаписать все»**, **«Пропустить конфликтующие»** или
  **«Отмена»**. Без явного выбора «перезаписать» локальная правка не трогается;
- перед подтверждённой перезаписью прежняя версия сохраняется в
  `.forgeide/import-backup/<timestamp>/`.

Сразу после импорта IDE спросит «Deploy the harness now?» — согласие нажимает
кнопку деплоя за вас (§5).

## 5. Обвязка: деплой и preflight

ForgeIDE работает по нативной модели forge-обвязки (T41): эталон
`.gigacode/hooks/settings.hooks.json` с плейсхолдерами `${PROJECT_ROOT}`/`${PYTHON}`
— источник правды, а рантайм (gigacode) читает **резолвнутый** `.gigacode/settings.json`.

**Deploy harness** делает по шагам:

1. приводит раскладку к forge-виду (если эталон лежит в корне `.gigacode/` от старой
   версии ForgeIDE — переносит его под `hooks/`);
2. снимает hash-манифест обвязки (baseline) — эталон под `hooks/` + `skills/`;
   `settings.json` не хешируется (он производный);
3. **генерит `.gigacode/settings.json`**: запускает резолвер самой обвязки
   `hooks/resolve_hook_paths.py` (разворачивает плейсхолдеры), а если его нет —
   встроенный `resolve.py`;
4. прогоняет preflight: собственный `hooks/preflight.py` обвязки, если он есть,
   иначе встроенный. Прогон стартует только если preflight прошёл; иначе — причина
   `HARNESS_PREFLIGHT`.

На карточке проекта — раздел **Harness**: кнопка **Deploy harness** и статус:

- зелёным — `deployed, preflight passed (baseline <когда>)`. Для forge-обвязки
  «passed» включает и состояние «задеплоено, но проект ещё не инициализирован»
  (forge-preflight exit 2) — энфорсмент включён, инициализация конфига идёт при
  первом прогоне пайплайна;
- красным — `preflight FAILED`, а под кнопкой — список проблем (для forge-обвязки —
  её `errors`/`init_needed`/`warnings`; для встроенного preflight — построчно).

Если прогон встал с причиной `HARNESS_PREFLIGHT`, run view показывает баннер
с подсказкой и кнопкой **Go to project card** — задеплойте обвязку там и
запустите прогон заново (сам остановленный прогон не резюмируется).

После деплоя работает контроль дрейфа: изменение любого файла обвязки вне
доверенного редактора IDE останавливает прогон перед следующей агентной фазой
(`harness-drift`) и показывает diff — принять новую версию или откатить решает
человек. Правка через редакторы IDE (§7) дрейфом не считается: она сама
становится новым baseline.

## 6. Канвас и конструктор пайплайна

**Open canvas** открывает конструктор с вкладками:

- **Canvas** — граф плиток. Палитра слева: перетащите тип шага на канвас;
  ребро между плитками (= `depends_on`) — протяжкой от коннектора одной плитки
  к другой. **Undo/Redo**, **Reset view**; правый клик по плитке — **Delete** /
  **Duplicate**. Ctrl/Cmd-клик выделяет подграф — его можно сохранить в
  библиотеку (**Save to library…**). Плитка с ошибкой валидации подсвечена,
  список ошибок — в панели деталей; ошибки уровня пайплайна — баннером сверху.
- **YAML** — тот же пайплайн текстом: **Preview diff** и **Apply to canvas**
  для правки руками.
- **Library** — библиотека плиток и подграфов: сохраняйте настроенную плитку и
  вставляйте её в другие пайплайны — id и пути промптов при вставке
  переименовываются сами, без ручной правки.

**Save pipeline.yaml** пишет файл через каноничный writer: перед сохранением
покажет diff (комментарии в существующем YAML не сохраняются — FR-2.7).

⚠ Заполняйте `question` у каждого гейта: канвас сегодня позволяет сохранить
гейт с пустым вопросом, но такой файл не пройдёт парсер при следующем открытии
(см. [Известные ограничения](#известные-ограничения)).

## 7. Редактирование плиток

Клик по плитке → панель деталей с редакторами:

- **Config** — поля шага (expects, allowed_write, env_scope, budget, retry…).
- **Prompt** (agent-плитки) — редактор с подсветкой markdown; **Preview render**
  показывает «Prompt preview (next run)» — промпт с подставленными параметрами,
  **Diff with git HEAD** — отличие от закоммиченной версии.
- **Judge-скрипт** — редактор python; **Прогнать судью (dry-run)** исполняет
  проверку немедленно, без прогона — удобно отлаживать `check_*.py`.

Правки файлов под `.gigacode/` через эти редакторы — «доверенный путь»
(баннер «Trusted path — save updates the harness cache + hash-manifest»):
они аудируются (`prompt.edited` / `harness.edited`) и обновляют baseline, а
не считаются дрейфом.

Важно про запущенный прогон: промпты снапшотятся на старте прогона. Правка
промпта во время прогона повлияет на шаг только через **Retry step** — тогда
шаг перечитает файл.

## 8. pipeline.yaml — краткий справочник

```yaml
version: 1
id: my-pipeline

params:
  feature_name: {required: true, hint: "..."}

steps:
  - id: my-phase
    type: agent
    runtime: gigacode              # имя из Runtimes проекта
    prompt: prompts/my-phase.md    # шаблон промпта, ${param} подставляются
    expects: ["docs/${feature.slug}/out.md"]   # артефакты, без них шаг FAILED
    allowed_write: [docs/**]       # глобы; запись вне — инцидент SCOPE
    env_scope: [JIRA_TOKEN]        # какие секреты получит процесс (только эти)
    budget: {tokens: 2000000, wall_clock: 30m, output_mb: 256}
    retry: {stream: 1, script: 0}

  - id: check-phase
    type: judge
    target: my-phase
    check: {command: [python3, .gigacode/skills/x/scripts/check_out.py], timeout: 5m}
    llm: {runtime: gigacode, prompt: prompts/judge-phase.md}  # опционально, поверх check
    fail_policy: {max_iterations: 3}   # judge FAIL → повтор target, максимум N раз

  - id: approve
    type: gate
    depends_on: [check-phase]
    question: "Утвердить?"             # обязателен — не оставляйте пустым
    options: [Утвердить, Отклонить]
    show: ["docs/${feature.slug}/out.md"]
    risk: R1

  - id: which-way
    type: branch
    routes: {small: fix-inline, big: full-cycle}   # условие → id следующего шага

  - id: tdd-per-task
    type: per_task_loop
    task_plan: docs/${feature.slug}/task-plan.json # откуда берётся список задач
    template: [...]                                # вложенные шаги, повторяемые по задаче

  - id: deliver
    type: outward
    actions: [git_push, create_pr]     # git_push | create_pr | jira_comment | jira_transition
    env_scope: [GIT_TOKEN]
```

`judge` требует хотя бы один из блоков `check`/`llm`; вердикт LLM-судьи без
детерминированного чека не может гейтить `outward`-шаги. Полная семантика —
[SD §5–6](sd/system-design.md).

## 9. Секреты

Секреты живут в `~/.forgeide/secrets.json` (плоский JSON `{"ИМЯ": "значение"}`,
права 600, атомарная запись). UI-редактора пока нет — файл правится руками.
Фаза получает **только** ключи, перечисленные в её `env_scope`; пустой scope —
пустое окружение.

Маскирование: значения секретов из `env_scope` фазы, а также типовые паттерны
(`Authorization: Bearer …`, `token=…`, `ghp_…`/`glpat-…`) заменяются на `***`
во всех **доверенных** файлах прогона до записи на диск — в вердиктах судей,
`accumulated_errors`, payload'ах аудита, `meta.json`. **Сырые** логи фаз
(`ground/ai-logs/…/stdout.jsonl`, `stderr.log`) по спеке не маскируются — они
untrusted и остаются как есть; именно поэтому `ground/ai-logs/` автоматически
держится в `.gitignore` (§4). Практическое правило: секрет, который нельзя
показать в сыром логе на вашей машине, фазе не выдавайте.

## 10. Запуск прогона

**Start run** → диалог запуска: **Feature slug** (например ключ Jira),
проверка readiness (валидный `pipeline.yaml`, заполненные обязательные params —
проблемы перечисляются прямо в диалоге) → **Start**.

Если git-дерево проекта грязное, прогон стартует, но в шапке run view появится
баннер «Run started on a dirty git tree…» и в аудит уйдёт `run.dirty_tree`:
scope-diff для уже грязных файлов ослаблен (§11). Рекомендация — запускать на
чистом дереве.

Экран прогона (**Run view**):

- **Timeline** — шаги со статусами и итерациями; фильтр инцидентов показывает
  tamper/scope/drift-события; «jump to log» ведёт к логу шага.
- **Step log** — live-хвост stdout/stderr шага (буфер 10k строк), поиск.
- **Inspector** — конфиг и файлы выбранного шага; промпт и judge-скрипт можно
  править прямо здесь (§7).
- **Run history** — прошлые прогоны фичи, resume прерванного.

Кнопки: **Cancel run**, **Retry step** (повтор упавшего шага; перечитывает
промпт с диска), **Roll back extra changes** (откат файлов, записанных вне
`allowed_write` — с подтверждением списка), **Export…** (выгрузка прогона
zip-архивом: стейт, аудит, логи).

### Гейты

Когда шаг доходит до `gate`, всплывает диалог «Gate — <id>»: вопрос,
кнопки-варианты из `options`, вкладки **git diff** (что изменилось), артефакты
из `show`, **History**. Для рискованных гейтов кнопки заблокированы, пока не
отмечен чекбокс **«Я посмотрел diff»**. Тексты от модели в диалоге помечены
как UNTRUSTED — сверяйте с вкладками diff/артефактов, они читаются напрямую с
диска. **«Закрыть (решить позже)»** прячет диалог, прогон продолжает стоять —
плитка на канвасе вернёт диалог по клику. Ответ уходит в аудит вместе с автором.

### Вопросы модели

Если фаза вернула вопросы, шаг встаёт в `WAITING_INPUT` и открывается диалог
«Questions — <id>»: **«Отправить ответы»** (ответы уйдут шагу на следующей
итерации; все поля обязательны) или **«Закрыть (ответить позже)»**. Третий
раунд подряд (лимит FR-10.5 — не более 2) эскалирует вместо нового вопроса:
диалог с вариантами **«Открыть промпт в редакторе»** и **«Отмена»**. «Открыть
промпт в редакторе» закрывает диалог (шаг остаётся стоять, эскалация
переоткрывается кликом по плитке) и переключает Inspector на вкладку **Prompt**
целевой фазы — правка сохраняется trusted-путём (аудит `prompt.edited`).
Рядом — кнопка **«Повторить с новым промптом»**: отменяет текущую попытку и
сразу повторяет шаг, подхватывая только что сохранённый промпт.

### Судьи

`judge` исполняет детерминированный скрипт и/или LLM-судью. Скрипт запускается
из **кэша IDE**, а не из рабочей копии — фаза не может переписать проверяющий
её скрипт. Вердикт LLM-судьи всегда перепроверяется детерминированным чеком.
FAIL судьи → target-шаг уходит на новую итерацию с накопленными ошибками в
контексте (секреты в них замаскированы), до `max_iterations` — дальше
эскалация человеку: диалог с вариантами **«Повторить»**, **«Править промпт»**
(требует текст — что поменять), **«Сбросить цепочку»**, **«Отмена»** и
**«Override»** (принять результат вопреки судье; причина обязательна, уходит
в аудит).

### Внешние действия

`outward`-шаги (`git_push`, `create_pr`, `jira_*`) движок исполняет только
после явно пройденного гейта — модель не может «дотянуться» до кнопки push.
Риск действия сверяется с `auto_max_risk` профиля критичности проекта.

## 11. Что контролируется, а что нет

Гарантии движка (см. [SD §9](sd/system-design.md)):

- переходы/гейты/вердикты — вне досягаемости модели; аудит пишет только движок
  (hash-chain, `~/.forgeide`);
- после каждой агентной фазы: scope-diff (`allowed_write`, сдвиг HEAD),
  tamper-check манифеста; перед каждой фазой — drift-check обвязки;
- бюджеты (токены, wall-clock, объём вывода) обрываются в полёте; осиротевшие
  процессы фазы зачищаются.

Слепые зоны scope-diff:

- **грязное git-дерево ослабляет scope-diff** — файл, уже изменённый до старта
  прогона, агент может дописывать дальше незаметно (scope-diff сравнивает
  статус-коды до/после фазы, а не содержимое). Об этом предупреждают баннер и
  `run.dirty_tree` (§10) — прогон не блокируется. Рекомендация: чистое дерево;
- gitignored-пути и записи **вне репозитория** scope-diff не видит (`git status`
  их в принципе не показывает; полный обход дерева ради этого противоречил бы
  NFR-4);
- **сеть** внутри фазы IDE не ограничивает — это вторая линия (хуки CLI, SD §9),
  превенция на уровне ОС — заготовка `PhaseSandbox` ([T29], пост-MVP).

## 12. Разбор типовых остановок

| Симптом | Причина | Что делать |
|---|---|---|
| Прогон сразу `paused: HARNESS_PREFLIGHT` | обвязка не задеплоена или preflight не прошёл | баннер в run view → **Go to project card** → **Deploy harness** (§5); список проблем под кнопкой |
| `preflight FAILED: hook referenced … not found` | в `.gigacode/hooks/` нет скрипта, на который ссылается `settings.hooks.json` | переимпортировать обвязку (свежий импорт привозит скрипты хуков) или доложить файл руками |
| `STOPPED (harness-drift)` перед фазой | файл обвязки изменён вне IDE (возможно, прошлой фазой) | посмотреть diff; принять (станет baseline) или откатить из кэша |
| Шаг `FAILED (SCOPE)` | запись вне `allowed_write` или сдвиг HEAD | посмотреть список; **Roll back extra changes**; расширить глобы, если легитимно |
| Шаг `FAILED (TAMPERED)` | фаза трогала проекцию манифеста (control-plane) | инцидент; проекция уже восстановлена движком, разобраться в логе фазы |
| Шаг `FAILED (BUDGET)` | превышены токены/время | поднять `budget` шага или сузить промпт |
| Шаг `FAILED (STREAM)` | оборван поток агента; ⚠ так же выглядит превышение `output_mb` | Step log; если вывод огромный — поднять `output_mb` |
| Шаг `FAILED (ARTIFACTS)` | не создан файл из `expects` | смотреть Step log; поправить промпт → **Retry step** |
| Судья валит все итерации | реальный дефект или сломанный чек | **Прогнать судью (dry-run)** на текущем дереве, отладить скрипт |
| Прогон завис в `WAITING_GATE`/`WAITING_INPUT` | закрыли диалог «на потом» | клик по плитке на канвасе откроет диалог снова |
| IDE перезапустили посреди прогона | процесс был убит | Run history → выбрать прогон → resume |
| Пайплайн не открывается после правки | гейт с пустым `question` сохранён конструктором | открыть `.forgeide/pipeline.yaml` руками, заполнить `question` |

## Известные ограничения

| # | Ограничение | Детали |
|---|---|---|
| 1 | Конструктор позволяет сохранить гейт с пустым `question`, парсер такой файл не откроет | валидатор/парсер асимметричны; лечится ручной правкой YAML |
| 2 | Превышение `output_mb` репортится как `FAILED (STREAM)`, а не `FAILED (BUDGET)` | флаг капа не доезжает от ProcessRunner до движка ([T27] бэклог) |
| 3 | Секреты в **сырых** логах фаз не маскируются | by design (SD §6.2): `ground/ai-logs/` держится вне git; не выдавайте фазам лишнего (§9) |
| 4 | Сеть внутри фазы не ограничивается IDE | вторая линия — хуки CLI; превенция — [T29] пост-MVP |
